# 서비스 요구사항 결정 기록

이 문서는 DBML/OpenAPI 생성 전 숨길 서비스 자체에 대해 확정한 요구사항을 기록합니다.

## 핵심 사용 순간

- 숨길의 핵심 사용 순간은 사용자가 평소에 랜덤에 가까운 장소를 스와이프하며 선호도 데이터를 쌓고, 동행자들과 여행방을 만들 때 그 선호도를 활용해 일정을 함께 짜는 순간입니다.
- 사용자가 여행지를 먼저 정하고 스와이프하는 구조가 아니라, 평소의 스와이프 행동이 나중에 여행 계획에 도움이 되어야 합니다.
- 사용자가 각자 무엇을 좋아하는지 직접 물어보지 않아도, 서비스가 평소의 스와이프 반응을 통해 선호도를 파악해야 합니다.
- 선호도 추론에 사용하는 행동 데이터는 장소 스와이프 반응인 `LIKE`, `NOPE`, `SUPER_LIKE`로 한정합니다.
- `SUPER_LIKE`는 다른 사용자에게 추천한다는 의미가 아니라, 사용자가 본인 선호도를 강하게 어필하는 반응입니다.
- `SUPER_LIKE`는 `LIKE`보다 강한 선호도 가중치이며, 언젠가 꼭 가고 싶은 장소에 가까운 의미입니다.
- 스와이프 카드에는 내 팔로우 중 해당 장소를 좋아한 사용자를 아바타 수준으로 표시할 수 있어야 합니다.
- 장소 저장은 `SUPER_LIKE`한 장소에만 허용합니다.
- 장소 상세 조회, 일정 추가/삭제, 저장, 커뮤니티 반응은 V1 선호도 추론 점수에는 포함하지 않습니다.
- 지도 위에서 자유 드로잉과 경로 생성을 함께 사용할 수 있어야 합니다.
- 사용자가 명시적으로 저장한 지도 그림/도형은 여행방에 저장되어 다시 볼 수 있어야 합니다.
- 사용자가 지도 위에 선을 그리면, 서비스는 그 선과 가까운 도로를 인식해 실제 이동 가능한 경로로 보정해야 합니다.
- 이 기능은 단순 그래픽 드로잉이 아니라 스케치 기반 경로 생성 또는 map matching/snap-to-road 기능입니다.
- 스케치 기반 경로 보정 provider는 Mapbox Map Matching API입니다.
- V1에서 사용하는 Mapbox profile은 `mapbox/driving`과 `mapbox/walking`입니다.
- 경로 보정은 Mapbox API가 제공하는 기능 범위 안에서 처리하며, 자체 도로 네트워크 인식 알고리즘은 만들지 않습니다.
- Mapbox Map Matching 결과는 Mapbox 지도 위에 표시하는 것을 전제로 합니다.
- 일정 협업뿐 아니라 지도 드로잉 과정과 도로 보정 결과도 여러 멤버에게 실시간으로 동기화되어야 합니다.

## 백엔드 설계 영향

- 지역 기준 데이터는 법정동코드 기반으로 관리합니다.
- 법정동코드는 `geo.legal_regions.code` 10자리 값을 기준으로 하며, 여행방의 다중 지역 선택은 `trip.trip_regions.legal_region_code`로 연결합니다.
- 법정동 동기화 이력은 `geo.legal_region_sync_logs`에 남겨 원천 데이터 반영과 실패를 추적할 수 있어야 합니다.
- 선호도 수집은 설문 중심이 아니라 스와이프 반응 기반으로 설계합니다.
- 스와이프 최종 반응과 이벤트 로그는 선호도 추론의 유일한 입력입니다.
- 스와이프 후보는 특정 여행방/목적지에 종속되지 않는 전역 개인 선호도 수집 feed를 기본으로 합니다.
- 전역 스와이프 feed는 랜덤에 가까운 경험을 제공하되, 지역, 카테고리, 인기/비인기, 실내/야외 등 후보 다양성을 보장해야 합니다.
- 사용자가 이미 반응한 장소도 일정 기간이 지나거나 관광공사 원본/AI 태그가 변경되면 다시 노출할 수 있습니다.
- 재노출된 장소에 사용자가 다시 반응하면 최종 반응은 갱신하고 이벤트 로그는 누적합니다.
- 여행방 생성 후에는 멤버들의 누적 스와이프 선호도를 여행 계획과 장소 후보 정렬에 활용해야 합니다.
- 여행방의 여행지 추천 리스트는 경로 설계 화면에서 좌측 일정/경로 추가 패널에 표시합니다.
- 추천 리스트의 목적은 장소 탐색보다 바로 일정에 추가하는 액션입니다.
- 추천 장소가 어떤 멤버 선호도에 맞는지는 카드 우측 하단 프로필 아바타 정도로 가볍게 표시합니다.
- 여행방 안에서 멤버의 세부 선호 태그와 선호도 점수는 공개하지 않습니다.
- 다른 멤버에게 공개되는 선호도 정보는 추천 카드의 matched member 아바타 수준으로 제한합니다.
- 추천 리스트의 기본 후보 풀은 현재 보고 있는 지도 viewport 안의 장소입니다.
- 기본 추천 리스트는 현재 지도 viewport 안의 후보만 사용하고, 참여자들의 태그 매칭 점수를 중심으로 정렬합니다.
- 거리는 태그 매칭 점수의 보조 점수 또는 동점 처리 기준으로 사용합니다.
- 추천 리스트에는 별도 SUPER_LIKE 탭이 있습니다.
- SUPER_LIKE 탭은 여행방 참여자들이 평소에 SUPER_LIKE한 장소를 SUPER_LIKE 강도/수 기준으로 우선 보여줍니다.
- 참여자 태그 매칭은 서비스 내부 태그 사전을 기준으로 합니다.
- 서비스 내부 태그 사전은 `preference_tagging_policy.md`의 고정 whitelist만 사용하며, whitelist 밖 태그는 확정 태그로 저장하지 않습니다.
- 관광공사 장소 데이터의 카테고리, 제목, 설명, 주소, 부가 정보를 AI로 분석해 내부 선호 태그를 추출/매핑합니다.
- AI 태그 추출은 사용자에게 직접 노출되는 AI 경로 추천 기능이 아니라, 추천 리스트 정렬과 선호도 매칭을 위한 백엔드 enrichment 기능입니다.
- 추천 태그 계산은 `confidence`, `weight`, `preference_discrimination` 중심으로 설계합니다.
- 태그 추출에 사용하는 관광지 원천 DB는 서비스 production 장소 마스터와 분리합니다.
- 관광지 원천 DB는 배포 시 production 서비스 DB와 물리적으로 분리할 수 있으며, 계약 문서에서는 `tourism_source` namespace로 표현합니다.
- SSAFY/KTO 스타일 `sidos`, `guguns`, `contenttypes`, `attractions` 원천 데이터는 `tourism_source` schema에서 import/enrichment 전용으로 관리합니다.
- 추천 근거는 긴 설명보다 멤버 매칭 표시를 우선합니다.
- 지도 협업은 일정 아이템 변경뿐 아니라 자유 드로잉 과정과 도로 보정 경로 이벤트 모델을 추가로 검토해야 합니다.
- WebSocket/STOMP 이벤트는 itinerary 변경, map drawing preview, route snapping 결과 변경을 구분해야 합니다.
- DBML에는 최종 Mapbox snapped route만 영구 저장합니다.
- 최종 snapped route는 일차(day) 단위가 아니라 출발 일정 아이템과 도착 일정 아이템 사이의 route segment로 저장합니다.
- 저장된 지도 그림/도형은 `itinerary.map_drawings`로 관리하며, 실시간 preview stroke는 저장하지 않습니다.
- 경로 보정용 임시 원본 stroke는 영구 저장하지 않고, 실시간 preview 또는 Mapbox 요청 처리에만 사용합니다.
- Mapbox route snapping이 실패하면 경로를 저장하지 않고 사용자에게 다시 그리도록 안내합니다.
- 추천 장소 API는 경로 설계 화면의 좌측 추가 패널에서 사용되며, 현재 지도 viewport, 참여자 태그 매칭 중심 정렬, 보조 거리 점수, SUPER_LIKE 탭 정렬을 지원해야 합니다.
- 추천 장소 API는 장소별로 선호도 매칭된 멤버 ID/avatar 목록과 일정 추가에 필요한 외부 장소 참조를 반환해야 합니다.
- 추천 장소 API는 다른 멤버의 raw/normalized 선호도 점수나 세부 태그 가중치를 반환하지 않습니다.
- 추천 장소 API의 태그 매칭 점수는 AI가 추출/매핑한 내부 태그를 기반으로 계산합니다.
- 스와이프 feed API는 현재 사용자의 내가 팔로우하는 사용자 중 해당 장소를 `LIKE` 또는 `SUPER_LIKE`한 사용자 목록을 제한된 아바타 정보로 반환할 수 있어야 합니다.
- 팔로우 기반 스와이프 신호는 팔로우 사용자의 raw 선호도 점수나 세부 태그 가중치를 노출하지 않습니다.
- 관광지 이미지 구성은 `tourism_source.attraction_images`의 일반 관광지 이미지 4개와 연결 가능한 콘텐츠랩 공모전 수상작 사진 1개를 우선 조합합니다.
- 수상작 사진은 S3 호환 object storage에 저장하고, DB에는 source metadata와 object key만 보관합니다.
- 파일명/지역명 기반으로 관광지에 연결할 수 있으면 연결하고, 정확히 연결되지 않으면 지역 수준 또는 미연결 상태로 보관합니다.
- 같은 관광지 또는 지역에 수상작 사진이 여러 장이면 최신 수상연도 우선, 같은 연도에서는 수상 순위 우선으로 노출합니다.
- 연결 가능한 수상작 사진이 없으면 일반 관광지 이미지와 보유 이미지로만 구성합니다.
- 일정에는 실제 day 외에 "일차 미정" 그룹이 있어야 합니다.
- "일차 미정"은 꼭 가기로 확정했지만 day number가 아직 정해지지 않은 장소를 담는 특수 일정 그룹이며, 실제 여행 일차 순서에는 포함하지 않습니다.
- "일차 미정"의 아이템은 나중에 사용자가 실제 day로 옮기거나 AI에게 재배치 요청할 수 있어야 합니다.
- AI가 "일차 미정" 장소 배치를 요청받으면, 미정 장소를 실제 day로 옮길 수 있습니다.
- AI는 미정 장소를 배치하면서 기존 실제 day 안의 장소 순서를 동선에 맞게 조정할 수 있습니다.
- AI는 명시적 요청 없이 기존 실제 day에 배치된 장소를 다른 day로 이동하지 않습니다.
- 선호도 계산은 원본 스와이프 이벤트 로그와 현재 선호도 projection을 함께 사용합니다.
- cold-start 단계의 합성 스와이프 데이터는 50개 고정 페르소나 기반으로 생성하고, 실제 사용자 이벤트와 source/storage를 분리합니다.
- `user_preference_tag_weights`는 현재 선호도 상태를 빠르게 조회하기 위한 materialized projection입니다.
- 선호도 projection은 raw 누적 점수와 추천에 사용하는 capped normalized score를 함께 저장합니다.
- 추천 계산에는 raw score가 아니라 사용자별 normalized score를 사용합니다.
- normalized score는 특정 태그를 계속 좋아요하거나 활동량이 많은 사용자가 그룹 추천을 과도하게 지배하지 않도록 제한된 범위로 계산합니다.
- normalized score 공식은 `tanh(raw_score / scale)`을 기본으로 합니다.
- `scale`은 운영 튜닝 가능한 설정값으로 둡니다.
- V1에서는 시간 감쇠 기반 선호도 재계산을 사용하지 않습니다.
- 선호도 projection은 스와이프 반응 이벤트가 발생할 때 해당 장소의 태그만 반영해 갱신합니다.
- 원본 이벤트 로그는 projection 복구, 공식 변경 시 재처리, 감사 용도로 남기며 일반 추천 요청에서 매번 재계산하지 않습니다.
- 별도 선호도 snapshot 테이블은 두지 않고, 현재 선호도는 projection과 원본 이벤트 로그로 관리합니다.
- route snapping 요청은 출발/도착 일정 아이템, Mapbox profile, 입력 좌표, radiuses, tidy 사용 여부, provider 응답 confidence, distance, duration, tracepoints, geometry를 저장할 수 있어야 합니다.
- Mapbox 일반 좌표 요청은 2~100개 좌표 제한이 있으므로, 프론트 또는 백엔드에서 드로잉 stroke를 적절히 단순화/downsample해야 합니다.
- route snapping API는 `DRIVING` 또는 `WALKING` mode를 받고 각각 Mapbox `mapbox/driving`, `mapbox/walking` profile로 변환합니다.

## 커뮤니티와 커뮤니티 게시글

- 팔로우 기능을 지원합니다.
- 팔로우 관계는 사용자의 커뮤니티 피드, 프로필, 팔로우 기반 추천 신호에 사용할 수 있어야 합니다.
- MVP에는 완성된 여행방을 공개 게시글로 공유하는 기능을 포함합니다.
- 여행방 OWNER와 MEMBER 모두 커뮤니티 게시글을 발행할 수 있습니다.
- 커뮤니티 게시글은 발행자와 게시 시각을 기록해야 합니다.
- 커뮤니티 게시글에는 사진/미디어를 첨부할 수 있어야 합니다.
- 커뮤니티 게시글에는 해시태그를 붙일 수 있어야 합니다.
- MVP 커뮤니티 게시글 공개 범위는 `PUBLIC`, `UNLISTED`만 지원합니다.
- `PUBLIC`은 커뮤니티 피드에 노출될 수 있고, `UNLISTED`는 링크를 아는 사용자만 접근하는 공유 글입니다.
- MVP 커뮤니티 게시글 댓글은 댓글과 1단계 대댓글까지 지원합니다.
- 2단계 이상의 중첩 대댓글은 지원하지 않습니다.
- 커뮤니티 게시글 댓글/대댓글은 작성자와 게시글 발행자가 soft delete할 수 있습니다.
- soft deleted 댓글/대댓글은 tombstone으로 표시할 수 있어야 합니다.
- MVP 신고는 커뮤니티 게시글, 댓글, 대댓글을 대상으로 지원합니다.
- 신고는 구조화된 신고 사유 선택과 optional 상세 설명을 함께 받을 수 있어야 합니다.
- 신고 사유 예: 스팸, 부적절한 콘텐츠, 괴롭힘/혐오, 저작권/권리 침해, 기타.
- 동일 사용자의 동일 대상 반복 신고는 중복 생성하지 않도록 제한합니다.
- MVP 신고 처리는 관리자/모더레이터가 신고 큐를 검토해 숨김, 복구, 삭제 처리할 수 있어야 합니다.
- MVP에서는 신고 수 기준 자동 숨김이나 신고 즉시 숨김을 적용하지 않습니다.
- 모더레이션 처리자는 처리 상태, 처리 시각, 처리 사유를 기록해야 합니다.
- 커뮤니티 게시글은 원본 여행방과 게시 시점 snapshot을 함께 저장해야 합니다.
- 원본 여행방이 이후 수정되어도 공개 게시글의 snapshot은 유지합니다.
- 커뮤니티 게시글 snapshot에는 공개 표시용 일정, 경로, 장소 표시 정보, 작성자 표시 정보를 저장합니다.
- snapshot 경로는 snapshot item 간 route segment로 저장합니다.
- 커뮤니티 게시글에 포함되는 장소 정보는 관광공사 원본을 다시 조회하지 않아도 표시 가능한 최소 정보로 저장합니다.
- 커뮤니티 게시글 미디어는 `media.media_files`를 기준으로 관리하고, 게시글별 정렬과 caption을 저장할 수 있어야 합니다.
- MVP 커뮤니티 게시글 반응은 좋아요와 리트립을 지원합니다.
- 리트립은 커뮤니티 게시글 snapshot을 내 여행으로 가져오는 액션입니다.
- 리트립을 실행하면 커뮤니티 게시글 snapshot 전체를 기반으로 새 여행방을 생성합니다.
- 리트립은 기존 여행방에 snapshot을 병합하지 않습니다.
- 리트립으로 생성된 새 여행방은 리트립한 사용자 혼자 OWNER로 시작합니다.
- 리트립 이후 동행자 추가는 일반 여행방 초대 흐름을 사용합니다.
- 커뮤니티 게시글 좋아요는 로그인 사용자 기준 토글형 1회 반응입니다.
- 한 사용자는 한 커뮤니티 게시글에 좋아요를 1개만 남길 수 있고, 다시 누르면 취소됩니다.
- 공개 게시글 좋아요와 리트립은 V1 선호도 추론 점수에 반영하지 않습니다.

## 여행 기록

- `/record` 화면의 기록은 여행방 trip에 묶어 관리합니다.
- 여행 기록은 누가 올렸는지, 어떤 trip/day/place와 관련 있는지, 사진/영상 파일이 무엇인지 저장해야 합니다.
- 기록 업로드자는 여행방 멤버여야 합니다.
- 기록 사진/영상은 `media.media_files`를 기준으로 저장하고, `record.trip_record_media`로 기록 엔트리와 연결합니다.
- 기록은 여행방 멤버에게 보이는 사적 기록이며, 커뮤니티 공개는 커뮤니티 게시글 발행 흐름을 사용합니다.

## 협업 변경 이력과 Undo/Redo

- 일정, 지도 경로/그림 수정, 메모, 체크리스트, AI tool write는 모두 협업 write command로 취급합니다.
- 협업 write command는 현재 사용자의 열린 WebSocket 협업 세션 안에서 뒤로가기/앞으로가기, 즉 undo/redo를 지원해야 합니다.
- undo/redo stack은 사용자별 활성 WebSocket 세션 단위로 관리하며, 다른 사용자의 변경을 undo/redo할 수 없습니다.
- 사용자별 활성 WebSocket 세션의 undo stack과 redo stack은 각각 최대 5개로 제한합니다.
- 새로고침, 재접속, 다른 기기 접속, WebSocket 세션 종료 이후에는 이전 세션의 undo/redo action을 제공하지 않습니다.
- 저장 전 자유 드로잉 preview는 프론트엔드 로컬 undo/redo로 처리하고, 백엔드 undo/redo는 저장된 snapped route나 저장된 도형/경로 command에만 적용합니다.
- undo/redo는 DB 트랜잭션 롤백이 아니라 기존 변경을 되돌리거나 다시 적용하는 보상 명령으로 처리합니다.
- undo/redo도 권한, 여행방 멤버십, 현재 version, 대상 리소스 변경 여부를 검사하고, 성공 시 일반 협업 변경과 동일하게 version 증가, 이벤트 저장, STOMP broadcast를 수행합니다.
- undo 이후 새 write command가 실행되면 해당 세션의 redo stack은 비웁니다.
- undo/redo 대상이 이미 다른 사용자 또는 같은 사용자의 다른 세션에 의해 변경되어 충돌이 생기면 자동 undo/redo를 거절하고 사용자에게 현재 상태를 기준으로 다시 요청하게 합니다.
- MVP에서는 일정, 메모, 체크리스트 변경을 보여주는 별도 활동 기록 UI를 제공하지 않습니다.
- 협업 변경은 화면 상태와 WebSocket 실시간 반영으로만 보여줍니다.
- 내부 변경 이벤트와 감사 로그는 유지하되, 사용자용 활동 기록 화면은 V1 이후로 미룹니다.

## AI 챗봇과 Tool Calling

- AI 챗봇은 단순 Q&A뿐 아니라 여행방 상태를 읽고 제한된 서버-side action을 수행할 수 있어야 합니다.
- AI agent session은 여행방 단위의 공유 세션이며, 여행방 참여자들은 같은 AI 대화 내역과 AI tool 실행 결과를 볼 수 있어야 합니다.
- V1에서는 여행방당 공유 AI agent session을 1개만 둡니다.
- 별도의 주제별 AI session, 사용자별 private AI session은 V1에서 만들지 않습니다.
- AI 대화 원문은 전체 저장합니다.
- 모델 context에는 전체 대화 원문을 매번 넣지 않고, 최근 메시지, 오래된 대화 요약본, 현재 여행방 상태, 허용된 tool context를 조합해 넣습니다.
- 오래된 대화 요약본은 긴 여행방에서도 AI가 주요 결정과 진행 맥락을 유지하기 위한 압축 context로 사용합니다.
- MVP에서 AI tool 실행 결과는 공유 AI 대화에 자연어 답변으로만 표시합니다.
- MVP에서는 별도의 tool 실행 카드나 tool call JSON을 사용자 UI에 노출하지 않습니다.
- 백엔드는 tool call/result reference와 감사 로그를 저장해 이후 실행 카드 UI로 확장할 수 있게 합니다.
- AI agent session의 사용자 메시지는 요청자 user를 명확히 기록합니다.
- AI tool write가 실행되면 변경 command의 소유자는 AI 세션이 아니라 해당 tool을 요청한 사용자와 활성 WebSocket 세션입니다.
- AI 챗봇은 현재 여행방 데이터, 요청자 본인의 상세 선호도 데이터, 그룹 추천용 집계/추천 결과를 context로 사용할 수 있습니다.
- 현재 여행방 데이터에는 일정, 지도 경로/그림, 메모, 체크리스트, 여행방 멤버 기본 정보가 포함됩니다.
- 요청자 본인의 상세 선호도 데이터에는 본인의 스와이프 기반 선호도 projection, SUPER_LIKE 장소, 저장 장소가 포함될 수 있습니다.
- 다른 멤버의 raw 선호도 점수, 세부 태그 가중치, 스와이프 로그는 AI 챗봇 context에 직접 제공하지 않습니다.
- 그룹 추천은 백엔드 추천/search tool이 계산한 후보, matched member avatar/id, 정렬 결과 수준으로만 AI 챗봇에 제공합니다.
- AI 챗봇이 선호도 기반 추천 이유를 설명할 때는 matched member 아바타/이름 수준만 언급합니다.
- AI 챗봇은 다른 멤버의 세부 선호 태그, raw/normalized score, 태그 가중치, 스와이프 로그를 설명에 노출하지 않습니다.
- 예시 설명: "민지님과 현우님 선호도에도 잘 맞는 장소예요." 또는 "그룹 추천 점수가 높은 후보예요."
- 백엔드는 Spring AI tool calling을 사용해 AI가 호출할 수 있는 도구를 명시적으로 정의합니다.
- AI 챗봇의 V1 실행 범위는 일정 편집, "일차 미정" 장소 배치, 여행지 탐색, 메모, 체크리스트로 제한합니다.
- 예시 사용자 요청: "일정 순서 재배치 해줘", "성심당을 1일차 두 번째로 옮겨줘", "비 오는 날 갈 만한 여행지 찾아줘", "준비물 체크리스트 만들어줘".
- "여행지 찾아줘"는 우선 후보를 찾아 보여주는 read/search tool로 처리합니다.
- 사용자가 같은 요청 안에서 "일정에 넣어줘", "1일차에 추가해줘"처럼 명시적으로 추가 의도를 표현하고 후보가 하나로 해석되면, AI는 확인 없이 일정 추가 write tool을 실행할 수 있습니다.
- 추가 의도는 있지만 day/위치를 판단할 수 없으면, AI는 "일차 미정" 그룹을 만들거나 재사용해 확정 방문 장소로 추가합니다.
- 추가할 장소가 여러 개로 해석되고 사용자가 여러 장소 추가를 의도한 경우, AI는 "일차 미정" 그룹에 최대 10개까지 추가할 수 있습니다.
- 검색 후보가 너무 많거나 사용자의 의도가 검색인지 일정 추가인지 불명확하면, AI는 write tool을 실행하지 않고 후보 또는 필요한 선택지를 먼저 보여줍니다.
- AI 여행지 탐색은 현재 지도 viewport를 우선 검색 범위로 사용합니다.
- 사용자가 자연어로 지역을 명시하면 해당 지역을 검색 범위로 사용할 수 있습니다. 예: "부산에서 비 오는 날 갈 만한 곳 찾아줘".
- 사용자가 지역을 명시하지 않으면 현재 지도 viewport와 여행방의 현재 작업 맥락을 기준으로 검색합니다.
- MVP 메모는 여행방 전체 메모와 일차(day) 관리 메모만 지원합니다.
- MVP 메모는 특정 일정 아이템, 장소, 지도 객체에 직접 연결하지 않습니다.
- 메모는 scope별 단일 관리 문서로 취급합니다.
- 여행방 전체 메모는 여행방당 1개만 둡니다.
- 일차(day) 관리 메모는 각 실제 day와 "일차 미정" 그룹마다 1개만 둡니다.
- 메모 tool은 여행방 전체 메모와 일차(day) 관리 메모를 생성/수정할 수 있어야 합니다.
- 여행방 멤버는 메모를 soft delete할 수 있습니다.
- soft deleted 메모는 삭제자와 삭제 시각을 기록합니다.
- MVP 체크리스트는 여행방 전체 체크리스트와 일차(day) 관리 체크리스트만 지원합니다.
- MVP 체크리스트는 특정 일정 아이템, 장소, 지도 객체에 직접 연결하지 않습니다.
- 체크리스트는 scope별 단일 리스트로 취급합니다.
- 여행방 전체 체크리스트는 여행방당 1개만 둡니다.
- 일차(day) 관리 체크리스트는 각 실제 day와 "일차 미정" 그룹마다 1개만 둡니다.
- MVP 체크리스트 항목에는 담당자를 지정하지 않습니다.
- 체크리스트 항목은 여행방 멤버가 함께 보는 공통 할 일로 취급합니다.
- 체크리스트 항목 자체에는 전역 완료 여부를 저장하지 않습니다.
- 각 체크리스트 항목은 멤버별 완료/미완료 상태를 관리해야 합니다. 예: "여권 챙기기" 항목에 대해 사용자별로 완료 여부와 완료 시각을 표시할 수 있어야 합니다.
- 신규 멤버가 합류하면 기존 체크리스트 항목은 기본적으로 미완료 상태로 보입니다.
- 체크리스트 tool은 여행방 전체 체크리스트와 일차(day) 관리 체크리스트를 생성, 수정, 정렬, 완료 처리할 수 있어야 합니다.
- 여행방 멤버는 체크리스트 항목을 soft delete할 수 있습니다.
- soft deleted 체크리스트 항목은 삭제자와 삭제 시각을 기록합니다.
- AI tool은 권한, 여행방 멤버십, itinerary version, no-confirmation tool policy를 통과한 경우에만 실행합니다.
- tool 실행 결과는 REST 일정 변경과 동일한 version 증가, 변경 이벤트 저장, STOMP broadcast 흐름을 따라야 합니다.
- V1 AI tool은 실행 전 확인 팝업 없이 즉시 실행합니다.
- read/search tool은 확인 없이 실행합니다.
- write tool은 명시적인 사용자 요청이 있고, 현재 세션의 공통 협업 undo/redo로 되돌릴 수 있는 작업만 허용합니다. 예: 일정 순서 재배치, 일정 아이템 day 이동, 메모 생성/수정, 체크리스트 생성/수정/완료 처리.
- 삭제, 대량 변경, 공개 공유, 초대/권한/설정 변경, 되돌리기 어려운 외부 API 호출처럼 undo/redo만으로 안전성을 보장하기 어려운 작업은 V1 AI tool registry에 등록하지 않습니다.
- write tool이 실행되면 UI는 실행 결과와 함께 공통 협업 undo/redo action을 제공할 수 있어야 합니다.

## 여행방 사용자 소통

- 여행방 참여자끼리 소통하는 UI는 AI agent session과 별도로 제공합니다.
- 사용자 간 채팅/소통 메시지는 AI 챗봇 메시지와 다른 도메인으로 분리합니다.
- AI agent session은 여행방 상태 질의, 추천, tool 실행 기록 중심이고, 사용자 간 소통 UI는 멤버 대화 중심입니다.
- MVP 사용자 간 소통은 여행방 전체 텍스트 채팅만 지원합니다.
- MVP 채팅 메시지는 특정 일정 아이템, 장소, 지도 객체에 연결하지 않습니다.
- 스레드, 멘션, 리액션, 일정/장소 연결 메시지는 MVP 범위에서 제외합니다.
- 여행방 채팅은 저장형 메시지이며, 새 메시지는 WebSocket으로 여행방 참여자에게 실시간 전달합니다.
- MVP 채팅 메시지는 수정할 수 없습니다.
- 사용자는 본인이 보낸 채팅 메시지만 soft delete할 수 있습니다.
- soft deleted 채팅 메시지는 "삭제된 메시지입니다" 같은 tombstone으로 표시할 수 있어야 합니다.

## 알림

- MVP 알림은 인앱 알림만 지원합니다.
- MVP 인앱 알림 생성 이벤트는 여행방 초대 관련 이벤트로 제한합니다.
- MVP 초대 알림은 앱 가입자인 사용자를 직접 초대했을 때 초대받은 사용자에게만 생성합니다.
- 초대 링크/코드로 참여하는 흐름은 별도 인앱 알림을 생성하지 않습니다.
- 일정, AI tool, 채팅, 메모, 체크리스트 변경은 MVP에서 인앱 알림을 생성하지 않고 화면 실시간 반영으로 처리합니다.
- 이메일, Web Push, FCM, APNs 알림 발송은 MVP 범위에서 제외합니다.
- 알림 모델은 이후 이메일/푸시 채널을 확장할 수 있게 설계합니다.
