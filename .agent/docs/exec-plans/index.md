# 실행 계획 관리

복잡한 변경은 채팅 안에서만 계획하지 않고 `.agent/docs/exec-plans/active/`에 실행 계획으로 남깁니다. 완료된 계획은 `.agent/docs/exec-plans/completed/`로 옮기고, 반복되는 후속 작업은 `.agent/docs/quality/scorecard.md`에 반영합니다.

---

## 1. 실행 계획이 필요한 경우

- 두 개 이상의 화면 또는 공통 CSS/JS를 함께 바꾸는 경우
- 인증, 여행방, 지도, AI 추천, 커뮤니티처럼 사용자 플로우가 연결되는 경우
- 기존 문서와 실제 코드가 어긋나 있어 정리 범위가 커지는 경우
- 자동 검사, 스크립트, 품질 기준을 새로 추가하거나 강화하는 경우

## 2. 파일 이름 규칙

- 진행 중: `.agent/docs/exec-plans/active/YYYY-MM-DD-short-topic.md`
- 완료됨: `.agent/docs/exec-plans/completed/YYYY-MM-DD-short-topic.md`
- 파일명은 영어 소문자와 하이픈을 사용하되, 문서 본문은 한국어로 작성합니다.

## 3. 실행 계획 템플릿

```md
# 작업명

## 목표

- 사용자가 얻어야 하는 결과를 구체적으로 적는다.

## 현재 상태

- 관련 화면, 파일, 문서, 알려진 부채를 정리한다.

## 변경 범위

- 수정할 파일과 수정하지 않을 파일을 구분한다.

## 수용 기준

- 완료 판단에 필요한 화면, 접근성, 반응형, 문서 조건을 적는다.

## 진행 로그

- [ ] 맥락 조사
- [ ] 구현
- [ ] 문서 갱신
- [ ] 검증

## 결정 기록

- 중요한 선택과 이유를 날짜와 함께 남긴다.

## 검증 로그

- 실행한 명령과 결과를 적는다.

## 후속 작업

- 이번 변경에서 의도적으로 남긴 리스크나 부채를 적는다.
```

## 4. 운영 원칙

- 실행 계획은 작업을 돕기 위한 문서이며, 작은 단일 파일 수정에는 만들지 않습니다.
- 계획의 체크박스는 실제 진행과 맞춰 갱신합니다.
- 완료 후에는 검증 로그와 남은 리스크를 정리한 뒤 `completed/`로 이동합니다.
- 반복되는 리스크는 실행 계획에만 두지 말고 자동 검사 또는 품질 점수 문서로 승격합니다.
