# 도메인 개발 정책

이 문서는 숨길 제품 코드를 도메인 단위로 개발할 때 반드시 지킬 작업 순서를 정의합니다.
대상은 `frontend/`, `backend/`의 사용자 기능, API, 도메인 로직, persistence, 화면 흐름 변경입니다.

## 핵심 원칙

- 도메인 개발은 test-first로 시작합니다.
- 구현자는 코딩 전에 해당 도메인과 사용 흐름을 요청자에게 이해시켜야 합니다.
- 요청자가 이해하고 진행을 허용하기 전에는 제품 구현 코드를 작성하지 않습니다.
- 테스트 없이 구현부터 시작하지 않습니다.
- 테스트가 실패하는 이유를 확인한 뒤 구현합니다.
- 변경 완료 후에는 테스트 결과와 남은 위험을 공유합니다.

## 필수 순서

도메인 기능을 개발할 때는 아래 순서를 지킵니다.

1. 도메인 맥락 설명

   구현자는 먼저 어떤 도메인을 바꾸는지 설명합니다.
   예: `auth`, `trip`, `itinerary`, `community`, `media`, `preference`.

2. 사용자 흐름 설명

   구현자는 최종 사용자가 어떤 행동을 하고 어떤 결과를 기대하는지 설명합니다.
   API나 내부 기능도 화면/사용자 행동과 연결해 설명합니다.

3. 수용 기준 정리

   구현자는 완료 판단 기준을 짧게 정리합니다.
   예: 성공 조건, 실패 조건, 권한 조건, 에러 응답, DB 변경, 이벤트/알림 여부.

4. 테스트 계획 제시

   구현자는 먼저 만들 테스트를 설명합니다.
   테스트는 정상 흐름뿐 아니라 중요한 실패/권한/경계 조건을 포함해야 합니다.

5. 요청자 이해 확인

   요청자가 흐름과 테스트 계획을 이해한 상태에서 진행합니다.
   이해가 불분명하거나 요구가 충돌하면 구현 전에 질문합니다.

6. 테스트 먼저 작성

   구현자는 제품 구현 코드보다 테스트를 먼저 작성합니다.
   가능한 경우 테스트가 실패하는 것을 확인합니다.

7. 구현

   구현자는 실패한 테스트를 통과시키기 위한 최소 구현부터 진행합니다.
   범위를 넘는 리팩터링이나 다른 도메인 변경은 분리합니다.

8. 검증과 설명

   구현자는 테스트를 실행하고 결과를 공유합니다.
   어떤 테스트가 무엇을 보장하는지, 남은 리스크가 무엇인지 설명합니다.

## 백엔드 기준

백엔드는 CQRS-lite 구조를 기준으로 테스트를 먼저 둡니다.

- command 기능은 command handler 테스트를 우선 작성합니다.
- query 기능은 query service 또는 mapper/repository 테스트를 우선 작성합니다.
- DB가 필요한 기능은 Flyway migration과 MyBatis mapper 테스트를 함께 고려합니다.
- 권한/인증이 관련되면 controller/security 테스트를 포함합니다.
- 실패 응답은 RFC7807 Problem Details 형태까지 확인합니다.

권장 테스트 범위:

- domain model/policy 단위 테스트
- command handler 테스트
- query service 테스트
- MyBatis mapper 또는 repository 테스트
- controller slice/integration 테스트

## 프론트엔드 기준

프론트엔드는 사용자가 보는 흐름을 먼저 테스트합니다.

- 라우트 진입 조건
- API 성공/실패 상태
- loading/empty/error 상태
- 권한이 없는 사용자 흐름
- 주요 form validation
- 모바일/데스크톱에서 깨질 수 있는 핵심 UI 상태

## 예외

아래 작업은 이 정책의 test-first 강제 대상이 아닙니다.

- 문서만 수정하는 작업
- 주석, README, 하네스 인덱스 갱신
- 단순 설정 문구 변경
- 빌드 스크립트의 명백한 오타 수정

단, 위 작업이 사용자 기능이나 도메인 동작을 바꾸면 이 정책을 적용합니다.

## 금지 사항

- 테스트 계획 없이 도메인 구현 시작 금지
- 요청자에게 사용자 흐름을 설명하지 않고 구현 시작 금지
- 테스트 실패 원인을 확인하지 않고 구현 완료 처리 금지
- command와 query 책임을 한 서비스에 무분별하게 섞기 금지
- DB schema 변경 후 migration/test 누락 금지
- 권한/실패 케이스가 있는 기능에서 성공 케이스만 테스트 금지

## 완료 보고 형식

도메인 개발 완료 보고에는 최소한 아래를 포함합니다.

- 변경한 도메인
- 사용자가 이해해야 할 흐름
- 먼저 작성한 테스트
- 실행한 검증 명령과 결과
- 남은 리스크 또는 다음 작업
